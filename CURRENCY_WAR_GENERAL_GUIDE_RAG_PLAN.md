# 货币战争通用攻略与专属 RAG 建设规划

> 文档状态：需求与总体实施规划  
> 适用范围：`4.4-4.7` 内容周期、标准博弈、默认挑战当前最高难度  
> 不适用：超频博弈、周期外版本、未经验证的数值或机制

## 1. 要解决的问题

当前项目已经能够做两类事情：

1. 从运行时 JSON 中精确查询角色、羁绊、装备、投资环境和投资策略。
2. 当问题命中明确关键词时，完整加载白厄反甲、卡芙卡-海瑟音持续伤害、姬子-启行列车三份阵容 Skill。

这两类能力之间仍有空白。用户经常提出的并不是某个阵容的专属问题，而是以下类型的运营问题：

- 当前节点该存钱、升级还是刷新？
- 低生命、高经济、核心对子同时出现时如何止损？
- 奖励、补给、遭遇三类节点分别该怎样取舍？
- 第一、二、三位面的目标和资源优先级是什么？
- 当前阵容尚未确定时，哪些信号说明应该继续过渡、确定路线或转型？

这些问题需要的是跨阵容的通用运营框架，而不是把某一个阵容 Skill 强行套上去。因此，本规划的目标是建立一个专属的“货币战争通用攻略 RAG 库”，让 Agent 能在保留结构化精确事实与阵容 Skill 的前提下，检索相关的通用攻略片段。

## 2. 核心目标

完成后，货币战争 Agent 应能：

1. 对角色费用、站位、羁绊效果、装备属性、配方、投资环境和策略效果等精确事实，优先引用 JSON 结构化资料。
2. 对已覆盖的成熟阵容，继续完整加载对应 Skill，而不是仅靠向量检索找零散片段。
3. 对经济、节点、风险、过渡、止损、路线确认等模糊运营问题，从“通用攻略库”检索少量相关段落。
4. 明确区分【基础库】、【攻略 Skill】、【通用攻略 RAG】和【策略推导】四类依据。
5. 在资料不足、检索未命中或资料存在版本冲突时，明确说明不确定性，而不是让模型用预训练记忆补齐。

## 3. 明确不做的事情

本阶段不做以下内容：

- 不把 5 份运行时 JSON 转成普通 RAG 文档，也不以向量相似度代替角色、装备、羁绊的精确查询。
- 不删除现有阵容 Skill，不把 Skill 改造成只能通过 RAG 找到的碎片。
- 不自动抓取网页后直接写进知识库；所有玩家经验内容必须先经过人工整理和版本核验。
- 不接入超频博弈的节奏、节点、数值或策略。
- 不让模型根据“看起来合理”的泛化经验生成未被资料支持的具体数值、概率或阈值。

## 4. 最终知识架构

```text
用户输入 + 当前对局状态
          |
          v
货币战争请求识别
          |
          +--> 结构化 JSON 精确查表
          |     角色 / 羁绊 / 装备 / 环境 / 策略
          |
          +--> 阵容 Skill 路由
          |     白厄反甲 / 持续伤害 / 列车 / 后续阵容
          |
          +--> 通用攻略 RAG
          |     节点 / 经济 / 升级 / 刷新 / 风险 / 转型
          |
          v
货币战争证据包
          |
          v
LLM 根据证据给出本轮策略推导
```

### 4.1 四种知识的职责

| 类型 | 保存内容 | 调用方式 | 是否允许替代其他类型 |
| --- | --- | --- | --- |
| 结构化基础库 | 角色、羁绊、装备、环境、策略的明确事实 | 名称精确匹配或 `lookup_currency_war_data` | 不允许被攻略文字覆盖 |
| 阵容 Skill | 一套阵容的完整机制、路线条件、装备、过渡、转型和风险 | 关键词路由后完整加载 | 不用于所有通用问题 |
| 通用攻略 RAG | 跨阵容的节点、经济、升级、刷新、止损、路线判断框架 | 向量检索少量相关片段 | 不得生成或覆盖精确数值 |
| 策略推导 | 对当前对局的具体建议 | LLM 根据前三类证据推导 | 必须说明是推导，不能伪装为固定规则 |

### 4.2 为什么不把 Skill 改成 RAG

Skill 是人为组织好的“整套决策手册”。以白厄反甲为例，是否成型、装备组件、护盾来源、白厄星级、受击环境和收尾能力之间存在连续逻辑；把它切成零散片段后，RAG 可能只取到“反伤不能击杀”，却没取到“必须保留收尾”的上下文。

因此：

- 用户明确讨论某个已支持体系时，完整加载该体系 Skill。
- 用户问跨体系或未确定体系的运营问题时，检索通用攻略。
- 用户仅问一个实体的属性或效果时，只查 JSON，不需要 Skill 或 RAG。

## 5. 当前项目状态

### 5.1 已经存在的能力

- `data/currency-war/runtime/4.4/*.json`：运行时结构化资料，共 72 个角色、33 个羁绊、158 件装备、83 个投资环境、334 个投资策略。
- `src/main/currency-war/grounding/currency-war-facts.ts`：从用户文本识别已知实体并精确查询 JSON。
- `src/main/currency-war/grounding/currency-war-grounding.ts`：构建“货币战争本轮证据包”。
- `resources/skills/currency-war-*`：已有白厄反甲、卡芙卡-海瑟音持续伤害、姬子-启行列车 Skill。
- `src/main/rag/*`：已有 Ollama embedding、JSON 向量索引、关键词回退检索和 `search_knowledge` 工具。

### 5.2 当前缺口

现有 `search_knowledge` 默认加载的是 `resources/cyrene/knowledge` 下的 Cyrene 世界观资料，不是货币战争攻略。它不能作为游戏通用攻略的检索入口。

此外，现有 Skill 路由只识别 3 套体系的少量关键词。用户问“二位面低血该升级还是刷新”时，既不必然命中阵容 Skill，也没有专属的货币战争通用 RAG 可以调用。

## 6. 通用攻略库的内容设计

通用攻略不是一篇大文章，而是一组职责单一、可单独检索的 Markdown 文档。第一版创建以下 6 份核心文档。

### 6.1 `standard-node-flow.md`

目的：描述标准博弈的固定节点流程与每个节点的决策目标。

必须包含：

- 第一位面：`1-1` 奖励、`1-2` 奖励、投资策略、`1-3` 战斗、`1-4` 战斗、`1-5` 补给、`1-6` 战斗、`1-7` 遭遇、`1-8` 奖励、`1-9` 首领。
- 第二、三位面：战斗、投资策略、战斗、补给、战斗、遭遇、奖励、首领的固定顺序。
- 每类节点应收集的信息、可做的操作和不可做的错误判断。
- 明确节点顺序是硬规则；具体奖励内容、商店内容和战斗结果是局内变量。

### 6.2 `economy-and-tempo.md`

目的：给出跨阵容的经济、经验、人口、刷新和升星判断框架。

必须包含：

- 经济资源的作用：金币、经验、等级、商店刷新、角色星级、即时战力。
- “升级”与“刷新”的比较维度：下一战压力、生命、核心对子、目标费用、空位收益、当前等级商店概率、剩余节点。
- “存钱”与“止损”的比较维度：生命压力、下一节点类型、是否存在立即成型机会、经济环境、即将到来的首领。
- 只允许写条件化判断，例如“生命压力高且存在一张即可升星的核心对子时，刷新价值上升”；不能凭空写固定金币线或概率。

### 6.3 `reward-supply-encounter.md`

目的：描述奖励、补给、遭遇节点的通用选择优先级。

必须包含：

- 奖励节点：经济、可立即升星、核心装备组件、阵容关键角色、通用即时战力之间的比较方法。
- 补给节点：免费角色与装备如何按当前路线、角色费用、站位、装备需求和短期战力选择。
- 遭遇节点：风险、奖励、生命和首领距离之间的取舍。
- 所有“优先级”必须标记为经验性框架，并允许阵容 Skill 覆盖。

### 6.4 `combat-risk-and-stabilization.md`

目的：在阵容尚未成熟时帮助 Agent 判断战斗风险与止损。

必须包含：

- 影响下一战风险的可观测信号：生命、节点类型、前后台人数、角色星级、装备数量、已激活羁绊、商店对子、首领临近程度。
- “立即战力”与“长期上限”的区别。
- 何时不能为了经济或理论最优路线继续空过战力。
- 缺少敌方阵容、词缀或战斗结果时，必须把建议表述为条件判断。

### 6.5 `route-commitment-and-pivot.md`

目的：说明如何从“过渡阵容”判断是否应该确定体系、继续投入或转型。

必须包含：

- 路线确认信号：核心角色、可升星进度、关键装备或组件、关键羁绊、生命和经济是否健康。
- 路线放弃信号：关键部件长期未出现、核心无进度、生命危险、另一体系已有更高价值成型信号。
- 未有专属 Skill 的体系只能给通用路线建议，不得虚构具体阵容终局。

### 6.6 `investment-choice-framework.md`

目的：描述投资环境和投资策略的通用评估方法。

必须包含：

- 先从 JSON 查询具体效果，再使用该框架评估即时战力、经济、装备、角色池、风险和路线契合度。
- 不能将“高稀有度策略”直接等同于更好；必须考虑其代价和当前局面。
- 选择后需要重新评估路线，而不是把环境/策略当作不可改变的绝对承诺。

## 7. 每份攻略文档的固定格式

所有通用攻略文档必须使用 YAML frontmatter，示例：

```markdown
---
id: currency-war-general-economy-and-tempo
title: 货币战争通用运营：经济、升级与刷新
scope: general
game_versions: ["4.4", "4.5", "4.6", "4.7"]
mode: standard-gambit
difficulty: highest-available
evidence_level: mixed
last_verified: "2026-07-24"
sources:
  - kind: user_verified_rule
    title: CURRENCY_WAR_GAMEPLAY_RULES_FOR_AGENT
    locator: 固定节点与标准博弈说明
  - kind: community_guide
    title: 通用运营攻略
    url: https://www.bilibili.com/video/BV14BUnBZECm/
    verified_claims_only: true
---

# 经济、升级与刷新

## 适用前提

- 当前玩法为标准博弈，目标为当前可挑战的最高难度。
- 当前问题涉及金币、经验、等级、商店刷新、角色升星或下一战压力。
- 角色费用、商店出现概率和投资效果必须另行查询结构化 JSON，不从本文件推断。

## 可直接采用的判断框架

- 先比较下一战风险，再比较短期可获得的即时战力，最后比较长期经济损失。
- 存在“一张即可升星”的核心对子且生命压力高时，刷新价值上升；若升级能立刻增加关键前台或提高目标费用角色的合理出现条件，则需要并列比较升级收益。
- 没有可完成的关键对子、生命健康且下一节点不要求立即战力时，保留经济的价值上升。

## 需要结合 JSON 查询的事实

- 当前等级对应的角色费用出现概率。
- 当前商店和备战席角色的费用、星级、站位、羁绊和角色赋能。
- 当前投资环境、投资策略和已激活羁绊的明确效果。

## 不可当作硬规则的经验

- “任何情况下都应在某个固定金币数开始刷新”不可作为本文件结论。
- “所有三费核心都应在同一等级刷新”不可作为本文件结论。
- 没有当前敌方信息时，不能断言下一战一定失败或一定获胜。
```

字段含义：

- `scope`：固定为 `general`，避免与阵容 Skill 混淆。
- `game_versions`：资料适用版本；版本不明时不得写入 4.4-4.7。
- `mode`：固定为 `standard-gambit`。
- `evidence_level`：`official`、`structured`、`community_verified`、`mixed` 四选一。
- `sources`：每个关键结论必须可追溯到来源和页内位置。
- `verified_claims_only`：玩家攻略只能提取已交叉核验的结论，不能整篇照录。

## 8. 资料收集与审核规则

### 8.1 来源优先级

| 优先级 | 来源类型 | 可用于什么 | 不可用于什么 |
| --- | --- | --- | --- |
| A | 游戏内实际验证、用户确认的标准节点顺序、官方公告/说明 | 固定规则、模式边界、明确机制 | 未公开的精确概率 |
| B | BWIKI 等结构化资料站 | 实体文本、效果、配方、节点说明 | 未经交叉验证的运营结论 |
| C | 高质量玩家通用攻略、视频文字稿 | 条件化运营经验、常见错误、路线判断 | 固定数值、版本无标记结论 |
| D | 零散评论、帖子、短视频切片 | 仅作线索 | 不直接进入知识库 |

### 8.2 录入流程

1. 收集原始链接、作者、发布日期、游戏版本、玩法模式和涉及主题。
2. 将内容拆成最小可核验结论，而不是复制整段文章。
3. 将实体事实与当前 JSON 对照；冲突时以 JSON 和官方/游戏内资料为准。
4. 将运营经验改写为“条件 -> 建议 -> 原因 -> 例外”的形式。
5. 标注来源、版本、可信度和是否需要后续游戏内验证。
6. 由人工审核后才写入通用攻略 Markdown。

### 8.3 冲突处理

- 若玩家攻略与 JSON 的角色费用、装备效果、羁绊数值冲突：丢弃该实体说法，保留其不依赖错误事实的运营思路。
- 若来源属于超频博弈：不写入标准博弈语料。
- 若版本早于 4.4，且无法确认 4.4-4.7 仍适用：标记过期，不进入默认检索库。
- 若多个经验攻略结论不同：保留为条件化分歧，不写成唯一标准答案。

## 9. RAG 技术设计

### 9.1 建议的新目录

```text
resources/currency-war/
  prompts/
    system.md                         # 已存在：主提示词
  knowledge/
    general/
      standard-node-flow.md
      economy-and-tempo.md
      reward-supply-encounter.md
      combat-risk-and-stabilization.md
      route-commitment-and-pivot.md
      investment-choice-framework.md
    sources/
      source-registry.json
```

Skill 继续放在 `resources/skills/currency-war-*`。不要把 `knowledge/general` 移入 Skill 目录，因为它们不是某一套阵容的专属手册。

### 9.2 建议的新运行时模块

```text
src/main/currency-war/rag/
  currency-war-knowledge.ts          # 读取并校验通用攻略 Markdown/frontmatter
  currency-war-retriever.ts          # 构建专属知识库并检索
  currency-war-rag-policy.ts         # 判断本轮是否需要检索、生成检索查询
  currency-war-rag-format.ts         # 把命中文本格式化为证据包片段
```

这些模块不应复用默认的 `loadCyreneKnowledgeDocuments()`，因为默认库混有 Cyrene 世界观资料。货币战争必须拥有独立文档集合、独立向量索引文件和独立 metadata 过滤条件。

### 9.3 检索触发策略

首版采用“程序判断 + 可选工具补查”，不让模型独自决定是否有通用攻略可用。

当同时满足以下条件时，自动尝试检索：

1. 本轮文本或对局总结明显属于货币战争；并且
2. 用户在问操作、优先级、运营、过渡、升级、刷新、经济、奖励、补给、遭遇、风险、止损、转型、投资选择之一；并且
3. 问题不是只要求一个实体的固定属性。

下列情况不自动检索：

- “白厄是几费？”：只查 JSON。
- “巡海游侠 3 羁绊效果是什么？”：只查 JSON。
- “以牙还牙甲能不能走？”：加载白厄 Skill，并可按需要补充通用风险攻略。
- 与货币战争无关的话题：不检索。

### 9.4 检索查询构造

不能把用户原句机械送给 embedding 模型。应从以下信息生成一条独立、完整的问题：

```text
模式：标准博弈最高难度；节点：2-3 补给；生命：35；金币：20；
当前有两个 3 费核心对子，下一战压力高。
问题：此时应优先刷新完成核心升星、升级人口，还是保留经济？
```

查询必须保留：当前节点、资源、风险、目标动作、阵容是否已确定。它不应凭空加入角色机制或数值。

### 9.5 结果过滤与预算

- 仅检索 `scope: general`、`mode: standard-gambit`，且 `game_versions` 覆盖当前目标版本的文档。
- 首版返回最多 3 个片段；每片段保留标题、来源、版本、内容和相似度。
- 若向量检索不可用，回退到同一货币战争语料的关键词检索；不得回退到 Cyrene 世界观库。
- 若没有可靠命中，证据包要明确写“未命中通用攻略”，模型只能依据 JSON、Skill 和策略推导回答。
- 不将检索片段直接当作最终答案；LLM 仍需结合当前 GameState 和结构化事实。

### 9.6 证据包格式

新增部分应放在现有“基础库事实”和“已加载攻略 Skill”之后：

```text
### 通用攻略检索结果
- [经济、升级与刷新]
  来源：community_verified，适用：4.4-4.7 标准博弈
  内容：当生命压力高、且存在一张即可升星的核心对子时，刷新价值上升；
  但若升级能立即增加关键前台或触发更高费用核心的合理出现条件，需要比较下一战压力。

### 使用约束
- 通用攻略是条件化运营经验，不能覆盖结构化实体事实。
```

## 10. 具体实施阶段

### 阶段 0：冻结边界与来源台账

目标：先确定什么能进入资料库，避免以后用大量无来源文本污染检索结果。

产物：

- `resources/currency-war/knowledge/sources/source-registry.json`
- 六份通用攻略的空白模板和 frontmatter 规范
- 来源审核清单

验收：

- 每一条进入语料的结论都能追溯版本、模式和来源。
- 没有超频博弈内容。
- 没有把角色数值或装备效果复制到通用攻略中作为第二份事实库。

### 阶段 1：编写第一版通用攻略

目标：先用已验证的固定规则、用户资料和经过审核的玩家经验，完成六份小而精的文档。

产物：第 6 节列出的六份 Markdown。

验收问题：

- “1-5 补给节点应优先补什么信息？”
- “2-3 低血且有核心对子，升级和刷新如何比较？”
- “还没有确定阵容时，什么信号说明应继续过渡？”
- “投资策略应先看哪些维度？”

验收标准：每个回答能指出使用了哪份通用攻略，不出现未记录的固定数值。

### 阶段 2：建立专属货币战争 RAG

目标：让通用攻略可以独立向量化、持久化和检索，不污染现有 Cyrene 世界观 RAG。

需要修改：

- 新增 `src/main/currency-war/rag/currency-war-knowledge.ts`。
- 新增 `src/main/currency-war/rag/currency-war-retriever.ts`。
- 新增 `src/main/currency-war/rag/currency-war-rag-policy.ts`。
- 新增 `src/main/currency-war/rag/currency-war-rag-format.ts`。
- 在 `src/main/app/main.ts` 创建独立 embedding provider 与独立索引，例如 `currency-war-general-guide-vector-index.json`。
- 不修改默认 `resources/cyrene/knowledge` 的加载逻辑。

测试：

- 通用攻略 loader 能拒绝缺少版本、模式、来源的文档。
- 检索只返回 `standard-gambit` 和目标版本覆盖的片段。
- Ollama 不可用时，只回退到货币战争通用攻略的关键词检索。
- 不会返回 Cyrene 世界观文本。

### 阶段 3：接入货币战争证据包

目标：在恰当问题中自动检索，并与 JSON 和 Skill 一起交给模型。

需要修改：

- 扩展 `CurrencyWarGroundingBuilder` 的依赖，使其接收通用攻略检索器。
- 在 `currency-war-grounding.ts` 中按第 9.3 节的策略决定是否检索。
- 更新 `resources/currency-war/prompts/system.md`，新增【通用攻略 RAG】来源说明和冲突优先级。
- 如有必要，新增只读工具 `search_currency_war_guidance`，供模型在自动检索未覆盖的问题中补查；该工具必须只搜索货币战争语料。

测试：

- 精确属性问题不触发 RAG。
- 模糊运营问题自动带入检索片段。
- 白厄反甲问题同时带入结构化事实、白厄 Skill 和必要的通用风险片段。
- 非货币战争问题不触发任何货币战争 RAG。
- 检索失败时，回答约束仍然禁止编造。

### 阶段 4：建立评测集与人工复核流程

目标：衡量检索是否真的提升建议质量，而不是只增加上下文长度。

创建 `tests/currency-war/rag/fixtures/general-guidance-cases.json`，至少包含：

- 节点流程类问题。
- 经济与刷新类问题。
- 低血止损类问题。
- 未确定阵容的路线判断问题。
- 明确阵容体系问题。
- 超频博弈诱导问题。
- 版本外资料诱导问题。
- 无关问题。

每个用例记录：输入、预期命中文档、禁止出现的结论、必须声明的不确定性。

验收：

- 预期文档进入前 3 个结果。
- 不应命中时结果为空。
- 检索结果不包含错版本或超频资料。
- 人工抽查 20 个真实对局问题，记录“有帮助、无帮助、误导、资料不足”四类结果。

### 阶段 5：逐步扩展阵容 Skill

目标：在通用 RAG 稳定后，再把高频、逻辑复杂的阵容写成专属 Skill。

候选顺序应由真实对局测试决定。每新增一个 Skill，必须同时具备：

1. 清晰的触发关键词或后续语义路由策略。
2. 机制、装备、运营、转型、风险和证据等级六部分内容。
3. 与 JSON 的实体事实一致。
4. 至少一组路由测试和一组完整证据包测试。

## 11. 推荐的第一轮实施范围

不要一次性搜集全网攻略。第一轮只做“最小可用通用运营库”：

1. 固定节点流程。
2. 经济、升级与刷新。
3. 奖励、补给与遭遇。
4. 战斗风险与止损。

原因：这四类覆盖绝大多数“没有明确阵容关键词”的问题，并且与现有手动 GameState 输入方式最契合。路线确认、投资选择可以在第一轮检索效果稳定后加入。

## 12. 成功标准

项目满足以下条件时，认为“货币战争通用攻略 RAG 第一版”完成：

- 通用攻略与阵容 Skill、JSON 数据在目录和代码中彼此独立。
- 货币战争问题不会检索 Cyrene 世界观库。
- 模糊运营问题可以得到 1 至 3 条有来源、有版本和模式限制的通用攻略片段。
- 精确实体问题仍然走 JSON 直查，且不需要 RAG。
- 明确阵容问题仍能加载完整 Skill。
- 所有新增资料都标注来源、版本和标准博弈适用性。
- 当资料不足时，Agent 明确说“本地资料未记录”或“未命中通用攻略”，不编造具体规则。
- 单元测试、RAG 检索测试、货币战争证据包测试、类型检查和构建全部通过。

## 13. 需要持续维护的内容

货币战争在 4.4-4.7 内容周期内仍可能有调整。每次更新资料时：

1. 先更新结构化 JSON 与来源报告。
2. 检查通用攻略中是否引用了受影响的机制或数值。
3. 将不再适用的段落标记为过期，不让它参与默认检索。
4. 更新受影响 Skill 的证据文件和版本范围。
5. 重新运行评测集，重点检查版本冲突与超频规则泄漏。

通用攻略 RAG 的价值不在于“资料越多越好”，而在于每一段都能说明：它适用于什么条件、来自哪里、为什么能用于当前对局。

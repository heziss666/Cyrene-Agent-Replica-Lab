# 货币战争游戏玩法辅助 Agent：总体设计

日期：2026-07-22  
状态：已确认，等待实现计划  
目标项目：`Cyrene-Agent-Replica-Lab`

## 1. 目标与边界

本项目将从通用 Cyrene Agent 逐步收敛为《崩坏：星穹铁道》货币战争玩法的对局辅助系统。

系统面向新手和有经验的玩家：玩家在每个节点输入当前对局状态，系统给出本节点操作、候选选择的排序、阵容方向、风险说明，以及接下来两到三个节点的运营规划。系统不自动操作游戏，不规避游戏限制，也不把不确定的旧版本资料伪装成当前版本事实。

第一版采用手动输入局面。后续可增加截图识别，但截图识别只能作为状态输入适配器，不能改变规则、规划和推荐核心。

## 2. 总体原则

1. 保留现有 Electron、Agent Loop、会话、RAG、记忆、工具系统、MCP、运行追踪和任务调度；不因游戏功能而立即删除通用能力。
2. 游戏事实、策略经验、确定性规则和语言表达分层，不把它们混在 Prompt 中。
3. 规则引擎负责确定性结论和约束；LLM 负责理解自然语言、调用工具、比较方案、解释建议和处理不确定信息。
4. 当前版本不确定的数据必须带版本、来源和置信度；不能进入确定性计算。
5. 所有推荐都应给出：当前动作、原因、备选、风险、需要补充的信息和短期规划。

## 3. 推荐架构

采用“结构化数据 + 规则/评分引擎 + RAG + LLM”的混合架构。

```text
玩家手动输入局面
  -> 局面状态校验
  -> 规则引擎生成合法动作与硬约束
  -> 评估器和规划器生成候选方案
  -> 结构化查询取得角色/羁绊/装备/策略事实
  -> RAG 检索运营经验和阵容打法
  -> Agent Loop 调用上述工具
  -> LLM 生成可解释的推荐
  -> Electron 展示推荐并保存本局状态
```

### 3.1 不采用的首发方案

- 不把游戏知识直接写入 System Prompt：难以版本化、无法精确计算，也难测试。
- 不只依赖向量 RAG：相似文本检索不能可靠计算金币、人口、羁绊档位或商店概率。
- 不在第一阶段引入图数据库：现有实体 ID 和引用关系足以先完成轻量关系遍历。待羁绊、赋能、装备适配和机制数据足够完整后，再评估是否需要图数据库。

## 5. 数据层

桌面目录 `CURRENCY_WAR_PLAN/currency_war_baseline_v2_1` 是当前资料基线，不应成为 Electron 运行时依赖。首次导入后，项目内的 `data/currency-war/` 才是可复现的数据快照。

### 5.1 三层数据职责

| 层级 | 内容 | 是否允许缺失 | 是否由运行时读取 |
| --- | --- | --- | --- |
| `staging` | 原始网页、CSV、人工记录和导入前 JSON | 可以 | 否 |
| `canonical/v3` | 带证据、版本、审核状态和缺失原因的事实 | 可以，但须说明原因 | 通常否 |
| `runtime/4.4` | 已编译、适合当前版本计算的最小事实集 | 关键字段不可以 | 是 |

### 5.2 当前资料的导入策略

- 导入 V3 Canonical 结构和 Runtime 结构，而不是直接使用旧 CSV。
- 保留 `dataset_version`、`game_version_target`、实体 ID、来源证据和缺失字段。
- 83 个未确认当前效果的投资环境继续排除在确定性决策 Runtime 之外。
- 角色赋能、羁绊档位、经济规则、商店概率等 P0 缺口必须有独立的补录和校验流程。
- 导入脚本、校验脚本和 Runtime 编译器改为根据脚本自身路径定位数据，禁止依赖命令执行目录。

### 5.3 RAG 文档边界

`guides/` 只保存无法或不适合完全结构化的经验，例如：阵容运营思路、转型条件、低血量止损、装备优先级解释、常见误区。

角色费用、羁绊成员、装备效果、策略效果和版本状态优先使用结构化查询工具。RAG 可以解释“为什么”，不能替代精确事实或数值计算。

## 6. Prompt 与 Skill 的职责

### 6.1 Prompt

Prompt 只约束语言模型的行为，不存放容易过期的游戏事实。

- `system.md`：系统身份、禁止臆造规则、要求优先调用游戏工具。
- `recommendation.md`：输出必须包含当前建议、备选、理由、风险、两到三个节点规划和置信度。
- `uncertainty.md`：当前版本资料缺失、输入不完整或证据冲突时的表达规则。
- `templates/`：按对局阶段补充关注点，不包含具体数值。

### 6.2 Skill

Skill 是可复用的工作流程，不是游戏资料库。

- `currency-war-coach`：收到对局请求时，先读取局面、校验数据、调用规划工具和知识查询工具，再给出建议。
- `currency-war-state-audit`：检查金币、人口、血量、节点、队伍、备战区、商店和可选项是否足以判断；缺什么就说明缺什么。
- `currency-war-data-review`：供维护者使用，检查新增数据是否具备版本、来源、证据、审核状态及缺失原因。

## 7. 游戏领域模块

### 7.1 `data/`

负责读取 Runtime JSON、建立按 ID/别名/名称查询的实体索引，并完成角色、羁绊、装备、策略和环境之间的引用遍历。

### 7.2 `state/`

负责本局状态的类型、输入校验、持久化和版本迁移。它与现有长期记忆系统分离：长期记忆保存玩家偏好和学习情况；本局状态保存可变的对局事实。

第一版状态至少包含：

- 对局标识、游戏数据版本、当前位面/节点、节点类型。
- 当前血量、金币、人口、经验、可上场人数。
- 上场角色、备战区角色、角色星级、装备、站位。
- 已激活或接近激活的羁绊。
- 当前商店、刷新次数、可选择的投资环境/投资策略/奖励。
- 已选择的环境、策略、顾问和重要历史选择。
- 玩家目标阵容、当前遇到的问题和可接受风险。

截图识别未来应输出同一份 `MatchStateInput`；这样输入方式变化不会波及规则和规划模块。

### 7.3 `rules/`

负责确定性规则：经济变化、人口和经验、商店概率、购买/出售限制、羁绊激活、装备限制、节点流转和特殊规则。规则未补齐前，必须返回“当前无法精确计算”的诊断，而不是猜测。

### 7.4 `evaluation/`

负责把局面转换为可比较的指标，例如当前战力代理值、羁绊完成度、核心角色缺口、经济健康度、血量风险、阵容方向一致性、候选策略匹配度。

第一版使用清晰可解释的启发式评分，不声称它是胜率预测。后续积累对局日志后，再考虑训练或校准模型。

### 7.5 `planning/`

负责产生候选动作和短期计划：现在买什么、是否刷新、是否升级、是否存钱、选择哪项投资、保留哪些过渡角色、何时考虑转型。输出的是结构化 `RecommendationCandidate[]`，由 LLM 解释而非由 LLM 自行杜撰。

### 7.6 `knowledge/`

负责游戏专用 RAG、关系查询、证据回传和版本过滤。它不替代通用 `src/main/rag/`，而是在其上增加货币战争的数据适配与检索策略。

### 7.7 `tools/`

第一批给 Agent Loop 暴露的工具建议为：

- `get_match_state`：读取当前局面。
- `validate_match_state`：指出缺失或矛盾输入。
- `lookup_game_entity`：查询角色、羁绊、装备、策略、环境及证据。
- `evaluate_current_board`：给出当前阵容的结构化评估。
- `compare_available_choices`：比较商店、环境或策略候选项。
- `plan_next_nodes`：生成短期运营计划。
- `search_currency_war_guides`：检索经验型攻略片段。

## 8. Electron 与 IPC

新增独立的货币战争工作区，不把手动录入表单塞进当前聊天页面。

- Renderer 只负责展示、编辑局面和请求推荐。
- Preload 只暴露受限的 `window.cyrene.currencyWar` API。
- Main 通过 `register-currency-war-ipc.ts` 校验请求、调用领域 Runtime 并返回结构化结果。
- 推荐过程仍可复用现有 Agent 运行追踪和 Activity 面板。

## 9. 分阶段实现顺序

### 阶段 A：数据接入与最小领域骨架

导入现有 4.4 Runtime 数据；建立数据加载器、实体查询、导入校验和固定对局测试样例。

### 阶段 B：手动局面输入与本局状态

定义 `MatchState`；实现 IPC、状态持久化和 Electron 表单；先确保用户能完整、可重复地录入一局。

### 阶段 C：最小可解释推荐

实现状态校验、角色/羁绊查询、阵容缺口分析和候选选择排序。此阶段的推荐只覆盖资料足够支持的结论。

### 阶段 D：经济与运营规划

补录并校验经济、人口、商店概率和节点规则后，实现升级、存钱、刷新、搜牌、升星的短期规划。

### 阶段 E：游戏专用 RAG 与 Skill

导入经验型指南；实现游戏专用检索、`currency-war-coach` 工作流和统一的解释输出。

### 阶段 F：完整策略层

逐步补齐投资环境、羁绊档位、角色赋能、装备适配和版本事件；扩大推荐覆盖范围。

### 阶段 G：增强输入与长期优化

增加截图识别适配器、对局日志、玩家偏好、复盘和基于真实对局的评分校准。图数据库仅在关系查询复杂度证明现有索引不足时考虑。

## 10. 测试与验收

每个规则、状态校验和规划器都使用固定局面样例测试。测试不验证“LLM 文风好不好”，而验证：

- 不完整局面能指出缺失字段。
- 未确认版本数据不会被当成当前事实。
- 非法动作不会被推荐。
- 相同局面在相同 Runtime 数据下得到稳定的候选排序。
- 推荐包含证据、假设和风险。
- 资料版本更新后，旧对局仍能标明其使用的数据版本。

## 11. 当前 P0 数据缺口

在扩展到完整运营建议前，优先补齐：

1. 4.4 当前投资环境的完整效果。
2. 经济、升级、利息、商店概率和节点规则。
3. 羁绊档位、效果与例外规则。
4. 角色前台/后台赋能、星级收益和顾问条件。

这四类信息决定了 Agent 是否能对“升人口、存钱、搜牌、转型、选环境”给出可信建议。

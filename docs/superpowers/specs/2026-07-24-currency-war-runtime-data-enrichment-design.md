# 货币战争运行时数据补全设计

## 1. 背景

当前 `data/currency-war/runtime/4.4` 下有五份供 Agent 直接读取的数据：

- `characters.json`
- `bonds.json`
- `equipment.json`
- `investment_environments.json`
- `investment_strategies.json`

现有数据已经能够提供名称、费用、站位等基础索引，但详情字段明显不完整：

- 72 个角色的前台赋能、后台赋能和星级数据全部为空。
- 33 个羁绊的 `effects` 全部为空。
- 158 件装备中，150 件的 `stats` 为空，部分基础属性被混入 `effect` 文本。
- 83 个投资环境中有 33 个 `effect` 为空。
- 投资策略字段基本完整，但尚未经过 4.4 版本全量复核。

这些缺失会直接削弱 Agent 的检索能力，并可能诱使模型用训练知识猜测角色羁绊、赋能和装备效果。

## 2. 目标

本阶段需要建立一条可以重复执行、能够追溯来源的数据补全流程：

1. 补齐当前可从可靠来源核实的角色、羁绊、装备和投资环境详情。
2. 复核投资策略及所有实体的 4.4 版本基础信息。
3. 同步维护 canonical 完整数据和 runtime 轻量数据。
4. 不允许使用 LLM 生成或改写游戏数值。
5. 无法核实的内容必须明确标记，不为了消除空值而猜测。
6. 后续 4.4-4.7 赛季更新时可以重新运行同一流程。

## 3. 数据来源原则

来源优先级如下：

1. 米哈游官方玩法说明、版本扩充说明和调整公告。
2. 能够定位到具体页面和修订时间的 BWIKI 详情页。
3. 具有结构化页面、版本信息和可复核原文的第三方游戏数据库。
4. 其他第三方资料只能用于定位线索，不单独支持精确数值。

当来源冲突时：

- 官方来源优先于社区来源。
- 同级来源优先采用版本更新更晚、目标版本更明确的内容。
- 仍无法消除的冲突不得自动覆盖，而应写入审查报告。

BWIKI 内容采用 CC BY-NC-SA 4.0。项目需要提供数据来源和许可说明，并将程序代码许可与游戏资料许可分开说明。

## 4. 三层数据架构

### 4.1 在线采集层

新增独立的数据补全脚本，负责：

- 下载官方公告和详情页。
- 解析 MediaWiki 模板或结构化页面。
- 规范化页面中的 HTML 和 Wiki 标记。
- 应用官方版本修正。
- 输出来源、修订时间、抓取时间和解析状态。

解析过程必须是确定性的。LLM 不参与字段提取、文本改写或数值判断。

### 4.2 Canonical 完整数据层

`data/currency-war/canonical/v3` 继续作为完整事实库，保存：

- 结构化的完整字段。
- 字段级证据来源。
- 版本状态和复核状态。
- 缺失、冲突与解析失败原因。

详情补全后应移除已经解决的 `missing_fields`，保留真正未解决的问题。

### 4.3 Runtime 轻量数据层

`data/currency-war/runtime/4.4` 由 canonical 自动生成，供 Agent、规则引擎和 Electron 界面读取。

Runtime 只保存回答游戏问题所需的信息，不携带冗长的证据链、审查记录或抓取元数据。顶层 `source` 继续提供主要数据来源。

## 5. 字段设计

### 5.1 角色

每个角色补充以下信息：

- 前台赋能名称、摘要、标签和技能。
- 后台赋能名称、摘要、标签和技能。
- 各星级的生命增幅、基础前台强度、基础后台强度、速度增幅、基础治疗强度和基础护盾强度。
- 统一的推荐装备列表。
- 专家顾问状态。

角色推荐装备不区分前后台。若来源分别提供两组推荐装备，生成 runtime 时合并、去重并保持首次出现顺序。

建议的 runtime 结构：

```json
{
  "name": "吉尔伽美什",
  "cost": 2,
  "field": "前后台",
  "roles": ["输出"],
  "bonds": ["命运圣杯", "能量"],
  "empowerment": {
    "front": {
      "name": "窃用宝物的贼人啊，本王准了！",
      "summary": "根据小队装备数量，提高全体幸运一击伤害，自身获得更多。",
      "tags": ["天赋"],
      "skills": [
        {
          "name": "王律键",
          "tags": ["天赋"],
          "description": "..."
        }
      ],
      "shared": false
    },
    "back": {
      "name": "窃用宝物的贼人啊，本王准了！",
      "summary": "...",
      "tags": ["战技", "终结技", "天赋", "普攻"],
      "skills": [],
      "shared": false
    },
    "stars": {
      "1": {
        "生命增幅": "15%",
        "基础前台强度": 125,
        "基础后台强度": 125,
        "速度增幅": "10%",
        "基础治疗强度": 60,
        "基础护盾强度": 60
      }
    }
  },
  "recommended_equipment": ["永动机", "碎星斩舰刀"],
  "advisor": false
}
```

角色页面只有一套共用赋能且站位为前后台时，`front` 和 `back` 使用相同内容并设置 `shared: true`。前台角色的 `back` 保持 `null`，后台角色的 `front` 保持 `null`。

技能描述中的斜线成长数值保留来源原文，例如 `200%/300%/400%/500%`。解析器不根据上下文擅自拆分或改写这些数值。

### 5.2 羁绊

每个羁绊补充：

- `base_effect`：羁绊基础机制。
- `effects`：以激活人数为键的档位效果。
- `special_rules`：地块、后台星级倍率、独立羁绊等补充规则。
- `members`：羁绊成员。

示例：

```json
{
  "name": "巡海游侠",
  "category": "阵营",
  "members": ["乱破", "不死途", "黄泉", "波提欧"],
  "base_effect": "巡海游侠队员获得伤害增幅和速度增幅……",
  "effects": {
    "1": "8%伤害增幅，8%速度增幅。",
    "2": "16%伤害增幅，16%速度增幅。",
    "3": "28%伤害增幅，28%速度增幅。",
    "4": "30%伤害增幅，30%速度增幅；进入战斗时行动提前100%。"
  },
  "special_rules": [
    "1/2/3/4星后台巡海游侠分别提供1/3/6/12倍额外增幅。"
  ]
}
```

`effects` 继续只使用数字键，因此现有 `bond-analyzer.ts` 可以直接计算已激活档位和下一档人数。

成员优先由角色数据中的 `bonds` 反向生成，再与羁绊详情页交叉核对。

### 5.3 装备

每件装备补充：

- `stats`：基础属性。
- `effect`：特殊效果。
- `recipe`：合成材料。
- 可用时补充标签和推荐角色。

基础属性不得继续混在 `effect` 中。简易装备本来没有特殊效果时，允许 `effect: null`。

### 5.4 投资环境

补齐所有能够核实的 `effect`，并复核名称和版本。当前 33 个空效果应优先处理。

### 5.5 投资策略

投资策略当前没有明显的空字段。本阶段只做 4.4 版本复核、结构校验和来源更新，不进行无意义的文本重写。

## 6. 数据处理流程

```text
下载来源
  -> 解析页面模板
  -> 清理 Wiki/HTML 标记
  -> 应用官方版本修正
  -> 与现有数据合并
  -> 检查来源冲突
  -> 更新 canonical
  -> 生成 runtime
  -> 生成审查报告
  -> 运行测试和构建
```

网络采集需要支持：

- 单请求或低并发执行。
- 请求间隔。
- 超时。
- 对 HTTP 429、5xx 和 BWIKI 567 响应进行有限重试。
- 失败后保留旧数据。
- 所有数据在内存中完成解析和校验后统一写入。

不得在部分页面成功、部分页面失败时覆盖完整数据集。

## 7. 缺失与冲突分类

审查报告需要区分：

- `not_applicable`：该实体本来不存在这个字段。
- `source_missing`：来源页面没有提供。
- `source_conflict`：两个可信来源存在冲突。
- `parse_failed`：页面存在内容，但解析器未能识别。

报告至少包含：

- 每类实体总数。
- 成功补全数量。
- 正常空值数量。
- 待核验数量。
- 冲突数量。
- 失败页面及原因。

## 8. 校验规则

### 8.1 角色

- 角色名称唯一。
- 角色费用、站位、羁绊引用有效。
- 前台角色必须有前台赋能。
- 后台角色必须有后台赋能。
- 前后台角色必须有两侧赋能或明确的共用赋能。
- 同一角色所有星级属性列长度一致。
- 推荐装备必须引用已有装备。

### 8.2 羁绊

- 羁绊名称唯一。
- `effects` 至少有一个合法数字档位。
- 档位按数值递增。
- 羁绊成员必须引用已有角色。
- 角色的羁绊与羁绊的成员列表双向一致。

### 8.3 装备

- 装备名称唯一。
- 非消耗品应具有基础属性或特殊效果。
- 合成配方中的材料必须存在。
- `stats` 与 `effect` 不重复保存同一基础属性。

### 8.4 投资数据

- 投资环境名称唯一，并具有可核实效果或明确缺失原因。
- 投资策略名称唯一，稀有度、效果和位面字段有效。

## 9. 测试策略

新增离线解析测试，使用固定且精简的页面模板样本，不在测试期间访问网络。

测试覆盖：

- 单站位角色。
- 前后台分别拥有赋能的角色。
- 前后台共用赋能的角色。
- 不同数量星级列的角色。
- 普通羁绊和独立羁绊。
- 有特殊规则的羁绊。
- 简易、进阶、特权、钻石和消耗品装备。
- 有合成配方和无合成配方装备。
- 正常投资环境和缺失来源。
- 网络失败、解析失败与来源冲突。
- canonical 到 runtime 的投影。

完成后运行：

- 数据完整性检查。
- 数据补全脚本测试。
- Currency War 相关测试。
- 全部测试。
- TypeScript 类型检查。
- Electron 构建。

## 10. 验收标准

本阶段完成时应满足：

1. 72 个角色均完成详情页采集或具有明确失败记录。
2. 适用角色不再存在无原因的空赋能和空星级数据。
3. 33 个羁绊均具有可用档位效果或明确待核验原因。
4. 装备基础属性与特殊效果完成拆分，能够核实的配方已补齐。
5. 投资环境空效果全部得到补全或明确说明。
6. 投资策略完成 4.4 版本复核。
7. canonical、runtime 和来源记录保持一致。
8. Agent 可以通过现有事实查询工具检索新增详情。
9. 不存在由 LLM 猜测生成的游戏事实。
10. 所有相关测试、类型检查和构建通过。

# 货币战争提示词与对局命名设计

## 1. 目标

本次改动解决两个相互独立、范围明确的问题：

1. 对局选择框只显示对局名称，不再把当前节点（例如 `1-1`）拼进名称。
2. 新建对局时自动生成不重复且易读的名称。
3. 将 Agent 的核心系统提示词改为“货币战争标准博弈辅助教练”，不再以日常陪伴型角色设定作为主要任务。

本次只调整对局命名和系统提示词，不改变 GameState 数据结构、RAG 算法、记忆系统、工具实现或对局编辑器字段。

## 2. 对局命名

### 2.1 显示规则

“当前对局”下拉框中的每一项只显示 `game.name`。

节点信息仍然属于 GameState，并继续在对局编辑区和对局总结中显示，但不再与名称拼接。这样可以避免用户误以为 `1-1` 是名称的一部分。

### 2.2 自动命名规则

用户点击“新建”且没有主动输入名称时，服务层生成名称：

```text
对局 1
对局 2
对局 3
...
```

生成算法采用“最小可用正整数”：

- 已有 `对局 1`、`对局 2`，新建结果为 `对局 3`。
- 已有 `对局 1`、`对局 3`，新建结果为 `对局 2`。
- 删除 `对局 1` 后再次新建，可以重新使用 `对局 1`。

只有完全符合 `对局 N` 格式的名称参与编号占用。用户自定义的“测试阵容”“追击队”等名称不占用自动编号。

### 2.3 兼容与约束

- 不批量修改已有对局名称，避免擅自覆盖用户数据。
- 已有重复名称继续保留；自动创建的新名称不会与已有标准自动名称重复。
- 用户仍可将对局重命名为任意合法名称，包括与其他对局相同的名称。
- 对局数量上限继续保持 10 个。
- 名称长度和空名称校验沿用现有规则。
- 唯一命名逻辑放在主进程服务层，确保 Electron 界面、测试或未来其他入口创建对局时行为一致。

## 3. 系统提示词定位

### 3.1 核心身份

Agent 的主要身份改为：

> 面向《崩坏：星穹铁道》“货币战争”玩法的对局决策辅助教练。

它应根据用户手动粘贴的对局总结，判断当前局面、给出当下操作建议，并规划接下来 1 至 3 个固定节点。

表达以专业、清晰、可执行为主，同时保留轻量的 Cyrene 语气。角色语气不能压过策略内容，也不能影响事实准确性。

### 3.2 适用范围

第一版系统提示词明确限制为：

- 游戏版本：当前项目资料对应的 4.4 版本。
- 模式：标准博弈。
- 难度：默认最高难度。
- 不把超频博弈的机制、节奏或数值混入回答。

如果用户询问超出当前资料范围的内容，Agent 应说明范围或不确定性，不得自行编造。

## 4. Prompt 组成

新增独立的货币战争 Prompt 资源目录。核心系统提示词使用一份主文件，包含稳定且全局有效的行为规则。

每次对话的 System Prompt 按以下顺序组成：

1. 货币战争核心系统提示词。
2. 用户当前选择的表达风格。
3. 如用户刚切换风格，追加一次性的风格切换提醒。

旧的 Cyrene `identity.md`、`soul.md`、`tone-rules.md` 不再作为核心提示词自动注入，避免日常陪伴目标与游戏决策目标冲突。旧资源文件暂时保留，不在本次删除，以便后续评估是否还有复用价值。

风格选项只允许改变措辞、亲和程度和表达节奏，不得覆盖以下内容：

- 标准博弈和最高难度范围。
- 事实依据与不确定性规则。
- 建议输出结构。
- 工具使用要求。
- 禁止编造游戏数据的要求。

## 5. 核心行为规则

### 5.1 输入检查

用户粘贴对局总结后，Agent 先判断信息是否足够。关键缺失信息会显著改变决策时，应先指出缺失项，再给出带条件的建议。

重点检查的信息包括：

- 当前节点。
- 生命值、等级、经验和货币。
- 已选投资环境与投资策略。
- 上阵、后台和备战席角色及星级。
- 商店内容。
- 装备库存与分配。
- 当前羁绊和已解锁顾问。

不要求每次机械复述所有字段；只指出会影响本次判断的缺失内容。

### 5.2 信息可信度

回答时区分四类信息：

1. **硬规则**：项目明确记录的固定玩法规则。
2. **结构化事实**：角色、装备、羁绊、顾问、投资环境和策略等数据。
3. **攻略经验**：来源于攻略资料的阵容与运营经验。
4. **策略推断**：模型根据当前局面作出的判断。

不得把策略推断伪装成固定规则。遇到资料缺失、版本可能不一致或多种路线接近时，应明确说明不确定性。

### 5.3 RAG 与工具

当回答依赖具体角色、装备、羁绊、顾问、投资环境或投资策略时，优先使用可用的知识检索工具获得依据。

工具不可用或检索结果不足时：

- 可以基于用户提供的信息给出条件化建议。
- 必须说明哪些结论缺乏资料支持。
- 不得虚构刷新成本、伤害公式、概率、角色效果或装备效果。

系统提示词只规定何时检索和如何使用结果，不把大量易变化的实体资料直接写进 Prompt。

### 5.4 节点规划

Agent 使用项目已经确认的标准博弈固定节点序列：

- 第一位面：`1-1 奖励`、`1-2 奖励`、选择投资策略、`1-3 战斗`、`1-4 战斗`、`1-5 补给`、`1-6 战斗`、`1-7 遭遇`、`1-8 奖励`、`1-9 首领`。
- 第二位面：`2-1 战斗`、选择投资策略、`2-2 战斗`、`2-3 补给`、`2-4 战斗`、`2-5 遭遇`、`2-6 奖励`、`2-7 首领`。
- 第三位面与第二位面的节点类型分布相同。

规划必须结合当前节点，不得建议已经错过的选择，也不得把固定节点顺序说成随机流程。

### 5.5 输出结构

在用户请求对局建议时，默认按以下顺序回答：

1. **当前判断**：局面强弱、主要问题与路线判断。
2. **现在怎么做**：按优先级列出本节点可执行操作。
3. **为什么**：解释资源、战力、经济或阵容方面的依据。
4. **备选方案**：当商店、奖励或资源结果不理想时如何调整。
5. **后续规划**：规划接下来 1 至 3 个节点的目标。
6. **风险与缺失信息**：只列会改变决策的重要风险。

简单问题可以缩短结构，但建议必须清楚区分“立即操作”和“后续计划”。

## 6. 数据流

```text
用户在对局页填写状态
        ↓
用户生成并复制对局总结
        ↓
用户粘贴到任意聊天会话
        ↓
PromptComposer 组装货币战争 System Prompt
        ↓
Agent 判断输入完整度并按需调用知识检索工具
        ↓
Agent 根据固定节点序列输出当前操作与后续规划
```

本次不自动绑定对局与会话，也不自动把 GameState 注入消息。继续保留“用户复制总结并自行粘贴”的轻量交互方式。

## 7. 错误处理

- Prompt 资源缺失时沿用现有必需资源加载错误，不静默退回旧人格 Prompt。
- 自动命名只读取当前对局列表，不依赖渲染进程保存的临时状态。
- 达到 10 个对局时继续返回现有上限错误。
- 知识工具失败时，Agent 应降低结论确定性，不得假装检索成功。
- 用户输入与项目资料冲突时，指出冲突并询问版本或条件，不直接覆盖用户描述。

## 8. 测试范围

### 8.1 对局命名测试

- 第一次自动创建得到 `对局 1`。
- 连续创建得到不同名称。
- 编号存在空缺时复用最小空缺。
- 自定义名称不占用自动编号。
- 下拉框不再包含节点编号。
- 重命名、删除、数量上限等现有行为不回归。

### 8.2 Prompt 测试

- 组合结果包含货币战争核心身份和范围限制。
- 组合结果包含固定节点规则和默认输出结构。
- 组合结果不再包含旧日常陪伴型核心身份文件。
- 所有风格仍能正常加载。
- 风格切换提醒仍只在切换后的首轮追加。
- Prompt 文件缺失时测试能明确失败。

### 8.3 全量验证

完成实现后运行 TypeScript 类型检查、相关单元测试、全量测试和 Electron 构建，确认聊天、对局编辑与 Prompt 组合没有回归。

## 9. 非目标

本次不实现：

- 自动读取游戏画面或 OCR。
- 对局与会话自动绑定。
- 自动更新 GameState。
- 新的向量库、知识图谱或检索算法。
- 超频博弈支持。
- 语音和通话。
- 自动验证每条攻略结论的胜率。

这些内容应在基础游戏教练 Prompt 验证有效后再独立设计。
